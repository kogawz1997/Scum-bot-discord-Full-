Role-based platform surfaces

Overview of how the platform is structured across owner, tenant, and player experiences while staying compatible with the current runtime.

Owner / Tenant / Player

Surface responsibilities

Each surface uses the same product language but serves a different user with different risk and information density.

Owner Console

Dense, operational, global

  • System health, runtime posture, and tenant fleet visibility
  • Security center, audit feed, request anomalies, and incident stream
  • Guarded paths into global config, backup, and recovery
  • Command palette for owner-only navigation and actions
Tenant Console

Scoped, guided, safer

  • Server and delivery posture inside tenant scope
  • Shop, economy, wallet, and delivery recovery grouped logically
  • Scoped config patches with validation and confirm-first flow
  • Tenant alert stream and support-facing player activity
Player Portal

Simple, trusted, mobile-first

  • Wallet balance and clear transaction history
  • Inventory-style shop UI and order state clarity
  • Redeem flow, profile, and Steam-link guidance
  • Legacy community features moved out of the main journey
Design Language

Shared but role-appropriate

The redesign keeps a tactical dark SaaS tone without turning the platform into a flashy game-themed dashboard.

Typography

Barlow + IBM Plex Sans Thai + JetBrains Mono

Condensed headings for scanning, clean Thai-friendly body copy, and monospace for ids, statuses, and runtime data.

Status grammar

Green / amber / red / ice

Success, warning, danger, and info states stay consistent across every surface and alert pattern.

Layout

Panels first, ornament second

Low-shadow panels, clean borders, and strong grouping keep the UI professional and readable under load.

Compatibility

What stays intact

This redesign is front-end-first. Existing backend flows, authentication, tenant routing, and runtime behavior remain untouched.

Routes preserved

Owner, tenant, player, legacy admin, and legacy player routes keep their existing behavior.

Business logic preserved

Purchase, redeem, delivery, auth, and tenant logic continue to run through the same backend contracts.

Legacy access preserved

Detailed workbench flows remain available when operators need the older high-density tooling.
