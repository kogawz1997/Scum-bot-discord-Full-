1) ระบบ Config และ Secret management

ปัญหา

.gitignore ตอนนี้ ignore แค่ .env และ environment folders ทั่วไป แต่ ยังไม่ครอบคลุม .env.*, backup env, dumped secrets, local override files แบบชัดเจน ขณะที่ใน repo มี env examples หลายชุดและระบบพึ่ง config หนักมาก. ถ้าทีมเผลอ commit ไฟล์แปลก ๆ เข้าไปอีก จะเป็นเรื่องงี่เง่าที่แพงมากทันที.  

ต้องแก้
 • เพิ่ม ignore ให้ครบ:
 • .env.*
 • *.env.local
 • *.bak
 • *.backup
 • secrets/
 • artifacts/ ที่มีผลตรวจหรือ dump
 • เพิ่ม secret scanning gate ให้ fail PR/commit ตั้งแต่ก่อน merge
 • แยก config เป็น:
 • required env
 • optional env
 • production-only env
 • สร้าง docs/CONFIG_MATRIX.md ว่า env ไหนใช้กับ bot / worker / web / agent

สถานะที่อยากได้
 • ไม่มี secret leak จากการเผลอ commit
 • คนตั้งระบบใหม่รู้ทันทีว่าต้องใส่อะไรบ้าง
 • production config ไม่ต้องพึ่งการเดาแบบมนุษย์สายมู

 • bootstrap/discord-runtime.js
 • bootstrap/admin-web.js
 • bootstrap/scum-webhook.js

และแยก logic ต่อไปนี้ออกจาก entrypoint:
 • config load
 • dependency init

สถานะที่อยากได้

 • debug ง่าย

6) ระบบ Multi-tenant ยังเป็น application scope มากกว่า isolation จริง

ปัญหา

README ระบุว่ามี tenant, subscription, license, API key, webhook, quota, marketplace, analytics foundation แล้ว แต่ก็ยอมรับตรง ๆ ว่า multi-tenant ยังไม่ใช่ database isolation ต่อ tenant และตอนนี้เป็น application-level scope มากกว่า DB-level isolation.  

ต้องแก้
 • อย่างน้อยเพิ่ม tenant guard ทุกชั้น:
 • API layer
 • service layer
 • query layer
 • audit layer
 • ห้าม query ใด ๆ ที่ไม่มี tenant context
 • เพิ่ม test ประเภท:
 • cross-tenant read leak
 • cross-tenant write leak
 • audit contamination
 • quota contamination
 • ถ้าจะขายจริง ควรมี roadmap ไป:
 • schema-per-tenant
 • database-per-tenant
 • หรือ partition strategy ที่ชัด

สถานะที่อยากได้
 • ลูกค้า A ไม่มีวันเห็นข้อมูลลูกค้า B แม้โค้ดพลาดนิดเดียว
 • support team ไม่ต้องมานั่งไล่ผี data bleed ตอนตีสอง

10) ระบบ Quality gate ยังอ่อนเรื่อง formatting และ static quality

 • ถ้าไหว เพิ่ม TypeScript ทีละชั้น หรืออย่างน้อย JSDoc typing ใน service layer
 • ให้ CI fail เมื่อ format ไม่ผ่าน

สถานะที่อยากได้
 • โค้ดอ่านง่าย
 • review ง่าย

ต้องแก้

 • restore rehearsal test

สถานะที่อยากได้
 • test ไม่ใช่แค่ “ฟังก์ชันรันได้”
 • แต่ยืนยัน policy สำคัญของระบบได้ด้วย

12) ระบบ Runtime locking และ supervisor ควรเข้มขึ้น

ปัญหา
[15/3/2569 19:48] C Graphics: README ระบุเองว่า ห้ามเปิด delivery worker ซ้ำทั้งฝั่ง bot และ worker แปลว่าระบบมีความเสี่ยงเรื่อง duplicate runtime responsibility. ดีที่มี test และ runtime-related scripts หลายตัว แต่ข้อจำกัดนี้ควรถูก enforce ให้แข็งกว่า “ห้ามนะ”.

สถานะที่อยากได้
 • ไม่มี worker ซ้ำเงียบ ๆ
 • ไม่มีของออกซ้ำเพราะ process สองตัวคิดว่าตัวเองเป็นพระเอก
